# 実装スキル

詳細設計書とコーディング規約に従って実装を行う。FE・BE・フルスタックを問わず、このスキルが基本の実装コマンド。

> FE担当・BE担当で完全に分業するチームは `/fe-impl` / `/be-impl` を使用する。

## 入力パターン

- `/implement TASK-XXX` - 指定タスクのスコープに沿って実装（推奨）
- `/implement` - 未実装の次のタスクを自動検出して実装

## 前提確認

実装開始前に以下を確認する。不足があれば該当スキルの実行を案内する。

1. `docs/requirements/` 配下に要件定義書が存在すること
2. `docs/specs/` 配下に該当機能の仕様書が存在すること
3. `docs/designs/` 配下に該当機能の設計書が存在すること
4. `docs/tasks/TASK-XXX.md` が存在し、スコープと完了条件が定義されていること

## 入力ゲート

実行開始時に、該当する開発計画（`docs/plans/`）のステータスが `✅ 承認済み` であることを確認する。
`📝 ドラフト` の場合は、先にレビューを完了するよう案内して終了する。

> **例外:** ステータスヘッダーがない既存ドキュメントの場合は、ユーザーに確認の上、承認済みとして扱ってよい。

## ファイアウォール制約

`.claude/rules/firewall.md` のルールに従うこと。

## 実装ルール（厳守）

### 設計書の遵守

- **設計書に記載されたディレクトリ構成、ファイルパス、コンポーネント名、関数名をそのまま使用する**
- 設計書にないファイルを勝手に作成しない
- 設計書にない機能を勝手に追加しない
- 設計と異なる判断が必要な場合は、実装を止めてユーザーに確認する

### コード品質

- TypeScriptの型を厳密に定義する（anyを使わない）
- 仕様書に記載されたバリデーションルールを漏れなく実装する
- 仕様書に記載されたエラーハンドリングを漏れなく実装する

### 各ステップの確認

各実装ステップ完了時に以下を実行する。

```bash
yarn build
yarn lint
```

エラーがあれば修正してから次のステップに進む。

## 実装フロー

1. **規約の読み込み**
   - `.claude/skills/implement/conventions.md` が存在すれば読み込む
   - 存在しない場合は `.claude/skills/fe-impl/conventions.md` と `.claude/skills/be-impl/conventions.md` を確認し、存在するものを読み込む
   - FEファイル（コンポーネント、ページ、hooks等）はFE規約を、BEファイル（API、DB、サービス層等）はBE規約を適用する
2. `docs/tasks/TASK-XXX.md` を読み、スコープと完了条件を確認する
3. タスクファイルに記載された設計書のリンクから設計書を読む
4. 設計書の該当セクションを特定し、実装内容を把握する
5. `docs/knowledge/` を確認し、実装に関連するナレッジがあれば参照する
6. `docs/plans/` 内の該当プランのステータスを `🔨 実装中` に更新する
7. タスクのスコープに沿って実装する
8. 各ステップ完了時にビルド・リントを確認する
9. 全ステップ完了後、タスクの完了条件を確認する
10. **ドキュメント整合性チェックを実行する**（後述）

## 設計書との差異が発生した場合

実装中に設計書の内容で問題が見つかった場合：

1. **実装を一旦止める**
2. 何が問題で、どう変更すべきかをユーザーに報告する
3. ユーザーの承認を得てから変更する
4. `docs/designs/` の該当設計書にも変更を反映する
5. 必要であれば `docs/specs/` の該当仕様書も更新する

## ナレッジの蓄積

実装中に以下のような知見が得られた場合、`docs/knowledge/` にMarkdownファイルとして記録する。

- 環境固有の制約や回避策（ファイアウォール、ネットワーク等）
- パッケージの互換性問題や注意点
- ビルド・リント時のハマりどころと解決策
- 設計書にない実装上のTips（パフォーマンス、エッジケース等）

## ドキュメント整合性チェック

実装完了後、コードと仕様書・設計書が一致しているかを確認する。**このステップは必須。**

### 確認手順（doc-checker エージェントで並列実行）

カスタムサブエージェント `doc-checker`（`.claude/agents/doc-checker.md` に定義済み）を `Agent` ツールで呼び出す。
設計書チェックと仕様書チェックを**2つ並列に**実行する。

```
エージェント①: 「以下の実装ファイルと設計書を比較してください。
 実装ファイル: [実装したファイルパスの一覧]
 設計書: docs/designs/<feature-name>-design.md」

エージェント②: 「以下の実装ファイルと仕様書を比較してください。
 実装ファイル: [実装したファイルパスの一覧]
 仕様書: docs/specs/<feature-name>-spec.md」
```

両方の結果を受け取り、差異があればまとめて対応する。

### 差異が見つかった場合

| 差異の種類 | 対応 |
|-----------|------|
| 設計書のファイルパス・関数名と実装が異なる | 設計書を実装に合わせて更新 |
| 実装で型やインターフェースを追加・変更した | 設計書の型定義セクションを更新 |
| 仕様書のバリデーションルールやエラーメッセージと実装が異なる | 仕様書を更新（または実装を修正） |
| 仕様書にないエッジケース対応を追加した | 仕様書に追記 |

更新箇所には以下のコメントを残す:

```markdown
<!-- implement: YYYY-MM-DD TASK-XXX [変更内容の要約] -->
```

### 整合性チェック結果の報告

完了報告にドキュメント更新の有無を含める。

## 完了後の状態

- タスクのスコープが実装されている
- ビルドエラー・リントエラーがない
- 設計書にない余計なファイル・機能が追加されていない
- **仕様書・設計書が実装後のコードと一致している**
- 実装中に得た知見があれば `docs/knowledge/` に記録されている
- （ステータス更新は `/commit` 完了時に行う）
