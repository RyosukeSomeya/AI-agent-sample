# 詳細設計書生成スキル

仕様書を読み込み、機能全体の詳細設計書を1ファイルとして生成する。

## 出力先

`docs/designs/` ディレクトリに、仕様書単位で1ファイルとして出力する。

### ファイル命名規則

```
docs/designs/<feature-name>-design.md
```

- `<feature-name>`: 仕様書のファイル名に対応するkebab-case名
- 例: `docs/specs/kaizen1-github-tree-spec.md` → `docs/designs/kaizen1-github-tree-design.md`

## 入力パターン

- `/design docs/specs/<name>-spec.md` - フルパス指定
- `/design <name>` - 名前のみ指定（`docs/specs/<name>-spec.md` に解決）
- `/design` - `docs/specs/` 配下の仕様書を一覧し、対象を選択して生成

## 引数の解釈

`$ARGUMENTS` が `/` を含まない場合（名前のみ）、以下の順序でファイルを探す。

1. `docs/specs/<name>-spec.md` が存在すれば使用する
2. 存在しない場合は `docs/specs/` 配下で名前を含むファイルを検索し、候補を提示する

## 前提

- `docs/specs/` に該当する仕様書が存在すること。なければ先に `/spec` を実行するよう案内する

## 入力ゲート

実行開始時に、入力となる仕様書のステータスが `✅ 承認済み` であることを確認する。
`📝 ドラフト` の場合は、先にレビューを完了するよう案内して終了する。

> **例外:** ステータスヘッダーがない既存ドキュメントの場合は、ユーザーに確認の上、承認済みとして扱ってよい。

## 手順

1. 仕様書を `Read` で読み込む（引数なしの場合は `docs/specs/` をスキャンして選択）
2. **入力ゲートを確認する**（仕様書が `✅ 承認済み` であること）
3. 対応する `docs/requirements/` の要件定義書も参照する
4. 既存コードベースを確認し、既に実装済みの部分との整合性を取る
5. 仕様書に含まれる**すべての機能**について詳細設計を記述する
6. `docs/designs/<feature-name>-design.md` に保存する（ステータス: `📝 ドラフト`）
7. **レビューサイクルを実行する**（後述の「レビューサイクル」セクション参照）

## 生成ルール

- 仕様書に記載された機能をすべてカバーする。追加も省略もしない
- ファイルパス、コンポーネント名、関数名は具体的に記述する
- 技術スタックは仕様書・要件定義に記載されたものを使用する
- 既に実装済みのコードがある場合、そのコードとの整合性を保つ
- 実装順序を記述し、タスク分割の参考になるようにする
- **図解にはMermaid記法を使用する**（コンポーネント階層図、処理フロー図、依存関係図等）

## レビューサイクル

`development-cycle.md` の「レビューサイクル（全フェーズ共通）」に従い、以下を実行する。

### 1. レビューポイントの提示

生成した設計書の主要な内容を箇条書きで提示し、`AskUserQuestion` で承認を求める。

```
📋 詳細設計書のレビュー
─────────────────────
ファイル: docs/designs/<feature-name>-design.md

【主要な内容】
- ディレクトリ構成: <追加・変更ファイル数>
- コンポーネント: <主要コンポーネントの一覧>
- API: <エンドポイント数と概要>
- 実装順序: <ステップ数と概要>

この内容で承認しますか？
```

選択肢: 「承認する」「修正を依頼する」「再生成する」

### 2. 修正ループ（修正依頼の場合）

1. ユーザーのフィードバック内容を確認する
2. ドキュメントを修正する
3. 修正箇所の差分を提示する
4. 再度 `AskUserQuestion` で承認を求める

### 3. 承認後の処理

1. ドキュメントのステータスを `✅ 承認済み` に更新する
2. 次のステップを案内する

## 完了後の案内

レビュー承認後、以下のように案内する:

```
✅ 詳細設計書を承認しました: docs/designs/<feature-name>-design.md

次のコマンドでタスク分割を行えます:

  /dev-plan docs/requirements/<feature-name>-requirements.md
```
