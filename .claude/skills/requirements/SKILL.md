# 要件定義書生成スキル

プロジェクトのアイディア、対話メモ、または対話形式のヒアリングから、構造化された要件定義書を生成する。

## 出力先

`docs/requirements/` ディレクトリに出力する。

### ファイル命名規則

```
docs/requirements/<project-name>-requirements.md
```

- `<project-name>`: プロジェクト名を表すkebab-case名（入力から推定、または対話で確認）

## 入力パターン

3つの入力方法をサポートする。

### パターン1: テキスト入力

```
/requirements タスク管理アプリを作りたい。カンバンボードで...
```

引数のテキストをプロジェクトの概要として使用する。

### パターン2: ファイル指定

```
/requirements docs/memos/idea.md
```

引数がファイルパス（拡張子 `.md`, `.txt` など）の場合、そのファイルを読み込んでプロジェクトの概要として使用する。

### パターン3: 対話形式

```
/requirements
```

引数なしの場合、ヒアリングを通じて情報を収集する。

## 手順

### 引数あり（テキスト or ファイル）の場合

1. 入力を解析する
   - ファイルパスらしき文字列（`.md`, `.txt` 等の拡張子を含む、または `/` で始まるパス）ならファイルを `Read` で読み込む
   - それ以外はテキスト入力として扱う
2. 入力内容から要件定義書の各セクションに必要な情報を抽出する
3. 不足している重要な情報があれば `AskUserQuestion` で**最大2回まで**ヒアリングする
   - 必須: プロジェクトの目的、主要な機能
   - 任意（不足時にヒアリング）: 技術スタック、ユーザー種別、データの種類
4. 要件定義書を生成して `docs/requirements/<project-name>-requirements.md` に保存する（ステータス: `📝 ドラフト`）
5. **レビューサイクルを実行する**（後述の「レビューサイクル」セクション参照）

### 引数なし（対話形式）の場合

1. `AskUserQuestion` でプロジェクトの目的・概要をヒアリングする
   - 「どのようなプロジェクトですか？目的と概要を教えてください」
2. 回答を踏まえ、`AskUserQuestion` で主要な機能をヒアリングする
   - 「主要な機能を教えてください（複数選択可）」＋回答から推測した選択肢を提示
3. 必要に応じて `AskUserQuestion` で技術的な制約・好みをヒアリングする
   - 技術スタック、対象プラットフォーム、スコープ外の事項など
4. 要件定義書のドラフトを生成して `docs/requirements/<project-name>-requirements.md` に保存する（ステータス: `📝 ドラフト`）
5. **レビューサイクルを実行する**（後述の「レビューサイクル」セクション参照）

## 生成ルール

- ユーザーが述べた内容を忠実に構造化する。勝手に機能を追加しない
- 情報が不足しているセクションは「要確認」として明示し、仮の内容を併記する
- 技術スタックが指定されない場合は、プロジェクトの性質から妥当な候補を提案し、ユーザーに確認する
- サンプルアプリ・PoC・MVP などスコープが限定される場合は、対象外セクションを明確にする
- **図解にはMermaid記法を使用する**（システム構成図、データフロー、ユーザーフロー等）

## レビューサイクル

`development-cycle.md` の「レビューサイクル（全フェーズ共通）」に従い、以下を実行する。

### 1. レビューポイントの提示

生成した要件定義書の主要な内容を箇条書きで提示し、`AskUserQuestion` で承認を求める。

```
📋 要件定義書のレビュー
─────────────────────
ファイル: docs/requirements/<project-name>-requirements.md

【主要な内容】
- プロジェクト目的: <目的の要約>
- 主要機能: <機能の要約>
- 技術スタック: <スタックの要約>
- 要確認事項: <件数と内容>

この内容で承認しますか？
```

選択肢: 「承認する」「修正を依頼する」「再生成する」

### 2. 修正ループ（修正依頼の場合）

1. ユーザーのフィードバック内容を確認する
2. ドキュメントを修正する
3. 修正箇所の差分を提示する
4. 再度 `AskUserQuestion` で承認を求める

### 3. 承認後の処理

1. ドキュメントのステータスを `✅ 承認済み` に更新する
2. 次のステップを案内する

## 完了後の状態

- `docs/requirements/<project-name>-requirements.md` に要件定義書が生成されている
- ステータスが `✅ 承認済み` に更新されている
- ユーザーに次のステップ（`/spec` の実行）が案内されている

## 次のステップの案内テンプレート

レビュー承認後、以下のように案内する:

```
✅ 要件定義書を承認しました: docs/requirements/<project-name>-requirements.md

次のコマンドで仕様書を作成できます:

  /spec docs/requirements/<project-name>-requirements.md
```
