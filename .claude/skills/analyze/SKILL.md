# エラー分析スキル

エラーログやバグ報告を分析し、原因特定と修正方針を含むレポートをMarkdownファイルとして出力する。
出力ファイルは `/fix` コマンドの入力として使用する。

## ファイアウォール制約

`.claude/rules/firewall.md` のルールに従うこと。

## 入力

- `$ARGUMENTS`: エラーログ、エラーメッセージ、またはバグの説明

## 手順

### 1. エラー情報の収集

$ARGUMENTS の内容を解析し、以下を整理する。

- エラーメッセージの全文
- 発生箇所（ファイル、行番号）
- エラーの種類（ビルドエラー、ランタイムエラー、型エラー等）
- 再現手順（分かる場合）

### 2. コードベースの調査

- エラーに関連するソースコードを読む
- `docs/specs/` の該当仕様書で正しい挙動を確認する
- `docs/designs/` の該当設計書で意図された実装を確認する
- 関連する依存パッケージのバージョンや設定を確認する

### 3. 原因の分析

以下の観点で原因を絞り込む。

- 実装が仕様・設計と一致しているか
- 型の不整合はないか
- 非同期処理の順序に問題はないか
- 依存パッケージの互換性に問題はないか
- 環境依存の問題はないか

### 4. 分析レポートの生成

`docs/analysis/ERR-XXX.md` に保存する（XXXは連番）。**分析レポートはgit管理対象外（ローカル運用）。**

```markdown
# ERR-XXX: [エラーの要約タイトル]

## 発生日時
YYYY-MM-DD

## 関連タスク
TASK-XXX（該当する場合）

## エラー内容

### エラーメッセージ
```
[エラーログの全文]
```

### 発生箇所
- ファイル: [ファイルパス]
- 行番号: [行番号]
- 発生条件: [いつ・どの操作で発生するか]

## 原因分析

### 直接原因
[何が直接的な原因か]

### 根本原因
[なぜその問題が起きたか]

### 影響範囲
- [影響を受けるファイル・機能]

## 修正方針

### 推奨する修正方法
1. [具体的な修正ステップ]
2. [具体的な修正ステップ]

### 修正対象ファイル
| ファイル | 変更内容 |
|---------|---------|
| [パス] | [何をどう変更するか] |

### ドキュメント更新の要否
- 仕様書(docs/specs/): [要否と理由]
- 設計書(docs/designs/): [要否と理由]

## 検証方法
修正が正しいことを確認する方法。
```

### 5. レポートの報告

```
📊 エラー分析完了
─────────────────────
レポート: docs/analysis/ERR-XXX.md
原因: [原因の要約]
修正方針: [方針の要約]
─────────────────────

修正を実行するには:
  /fix docs/analysis/ERR-XXX.md
```

## $ARGUMENTS

- `/analyze TypeError: Cannot read properties of undefined` - エラーメッセージから分析
- `/analyze ドキュメント保存ボタンを押しても保存されない` - バグの説明から分析
