# コードベース分析（リバースエンジニアリング）スキル

既存のコードベースを分析し、プロジェクトの開発資産となるドキュメント（仕様書・設計書等）をリバースエンジニアリングで生成する。

生成されたドキュメントは以下の2つの役割を持つ:

1. **開発資産** — プロジェクトの仕様・設計の正式な記録としてチームで共有する
2. **スキル連携の基盤** — `/implement`, `/fix`, `/review` 等のスキルが参照するドキュメント

新規プロジェクトではなく、**既にコードが存在するプロジェクト**にClaude Codeのスキル体系を導入する際に使用する。

## 入力パターン

- `/analyze-codebase` - プロジェクト全体を分析
- `/analyze-codebase src/app/api/` - 指定ディレクトリに絞って分析
- `/analyze-codebase src/components/Editor.tsx` - 特定ファイル・機能に絞って分析

## 出力ドキュメント

以下のドキュメントを生成する。

| ドキュメント | 内容 | 用途 |
|-------------|------|------|
| 仕様書 | 機能一覧、画面仕様、API仕様、データモデル | 開発資産 + `/fix`, `/review` の正解基準 |
| 設計書 | ディレクトリ構成、コンポーネント設計、型定義、依存関係 | 開発資産 + `/implement`, `/fe-impl`, `/be-impl` の実装ガイド |
| ナレッジ | 技術スタック、環境固有の制約、設定の注意点 | 全スキルの参考情報 |

## 手順

### 1. 出力先の確認（必須）

**ドキュメントの出力先をユーザーに確認する。このステップは省略不可。**

まずプロジェクトのディレクトリ構成を確認し、既存の `docs/` ディレクトリや類似のディレクトリがあるか調べる。

その上で `AskUserQuestion` で以下を確認する。

```
📁 ドキュメント出力先の確認
─────────────────────────

分析結果のドキュメントを出力するベースディレクトリを指定してください。
指定したディレクトリ配下に以下のサブディレクトリを作成します。

  <ベースディレクトリ>/
    specs/        - 仕様書
    designs/      - 設計書
    knowledge/    - ナレッジ

どのディレクトリに出力しますか？
```

選択肢の例: 「docs/（推奨）」「doc/」「その他（入力する）」

> 既に `docs/` 等が存在する場合はそれを推奨候補にする。

### 2. 出力ディレクトリの最終確認（必須）

ユーザーの回答を受けて、**作成するディレクトリの完全なパスを明示し、再度確認する。**

```
📁 以下のディレクトリを作成します。よろしいですか？
─────────────────────────

  <ベースディレクトリ>/specs/
  <ベースディレクトリ>/designs/
  <ベースディレクトリ>/knowledge/

既にファイルが存在する場合は上書きしません。
```

選択肢: 「OK」「変更する」「キャンセル」

**OKが得られるまで次のステップに進まない。**

### 3. テンプレートの確認

スキルディレクトリ内にテンプレートファイルが存在する場合は `Read` で読み込み、ドキュメント生成時にその形式に従う。

| ファイル | 用途 |
|---------|------|
| `templates/spec.md` | 仕様書のテンプレート |
| `templates/design.md` | 設計書のテンプレート |

テンプレートが存在しない場合は、`/spec`, `/design` スキルのテンプレートに準拠した形式で出力する。

### 4. コードベースの全体把握

以下を分析する。

- ディレクトリ構成とファイル一覧
- `package.json` / `pyproject.toml` 等から技術スタック・依存関係を把握
- 設定ファイル（`tsconfig.json`, `next.config.*`, `.env.example` 等）
- DB スキーマ（`prisma/schema.prisma`, マイグレーション等）
- 既存のREADMEやドキュメント

### 5. 機能の識別と分類

コードを読み込み、機能単位で整理する。

- **画面・ページ:** App Router / Pages のルーティング構造から画面を列挙
- **APIエンドポイント:** API Routeの一覧、リクエスト/レスポンス型
- **データモデル:** DBスキーマ、型定義からエンティティ関係を整理
- **共通コンポーネント/ユーティリティ:** 再利用されている共通モジュール
- **外部連携:** 外部APIとの通信、認証フロー

### 6. 仕様書の生成

機能単位で仕様書を生成する。ステップ3でテンプレートを読み込んだ場合はその形式に従う。テンプレートがない場合は `/spec` スキルのテンプレートに準拠した形式で出力する。

出力先: `<ベースディレクトリ>/specs/<feature-name>-spec.md`

各仕様書に含める内容:
- 機能概要
- 画面仕様（該当する場合）: 構成要素、操作、表示条件、画面遷移図（Mermaid）
- API仕様（該当する場合）: エンドポイント、メソッド、リクエスト/レスポンス
- データモデル: 関連するテーブル・型定義、ER図（Mermaid erDiagram）
- バリデーションルール: コードから読み取れるバリデーション
- エラーハンドリング: コードから読み取れるエラー処理

> **注意:** コードに書かれている挙動をそのまま仕様として記述する。「あるべき姿」の推測は行わない。

### 7. 設計書の生成

機能単位で設計書を生成する。ステップ3でテンプレートを読み込んだ場合はその形式に従う。テンプレートがない場合は `/design` スキルのテンプレートに準拠した形式で出力する。

出力先: `<ベースディレクトリ>/designs/<feature-name>-design.md`

各設計書に含める内容:
- ディレクトリ構成（該当ファイル一覧）
- コンポーネント設計: コンポーネント階層図（Mermaid）、props、状態管理
- API設計: ハンドラ構成、処理フロー図（Mermaid シーケンス図）、エラー処理
- DB設計: スキーマ定義（`prisma/schema.prisma` への参照リンク）
- 型定義: 主要なインターフェース・型
- 依存関係: モジュール間の依存関係図（Mermaid）

> **DB設計にはスキーマのコピーを埋め込まず、`prisma/schema.prisma` 等への参照リンクのみを記載する。**

### 8. ナレッジの生成

分析中に得られた技術的な知見をナレッジとして記録する。

出力先: `<ベースディレクトリ>/knowledge/<topic>.md`

対象:
- 技術スタックとバージョンの組み合わせ
- 環境固有の設定・制約
- コードベース固有の規約やパターン
- 既知の技術的負債やワークアラウンド

### 9. CLAUDE.md の生成・更新

分析結果を基に、プロジェクトルートの `CLAUDE.md` を生成または更新する。

**既に `CLAUDE.md` が存在する場合:**
- 現在の内容を `Read` で読み込む
- 分析結果と照らし合わせ、**追加・更新すべき項目のみ**を提案する
- `AskUserQuestion` で更新内容を提示し、承認を得てから反映する
- 既存の記述（ユーザーが書いたルール等）は削除しない

**`CLAUDE.md` が存在しない場合:**
- 分析結果から新規生成する
- 生成内容を `AskUserQuestion` で提示し、承認を得てから書き出す

**CLAUDE.md に含める項目:**

```markdown
# <プロジェクト名>

## プロジェクト概要
<コードから読み取れるプロジェクトの目的・概要>

## 技術スタック
- Frontend: <検出したFWとバージョン>
- Backend: <検出したFWとバージョン>
- DB: <検出したDBとORM>
- パッケージマネージャー: <検出したパッケージマネージャー>

## コマンド
- `<dev command>` - 開発サーバー起動
- `<build command>` - ビルド
- `<lint command>` - リント
- `<test command>` - テスト

## コードスタイル
<コードから読み取れる規約・パターン>

## ディレクトリ構成
<主要ディレクトリの役割>
```

> `CLAUDE.md` は常にコンテキストに読み込まれるため、**簡潔に保つこと**が重要。詳細は設計書・仕様書・ナレッジに委ね、CLAUDE.md にはプロジェクト全体の概要と基本情報のみを記載する。

### 10. 開発スタイルの確認と規約ファイルの生成

`AskUserQuestion` で開発スタイルを確認し、対応する規約ファイルの雛形を生成する。

```
実装コマンドの使い方を選択してください。

・フルスタック・分業なし → /implement を使用
  （FE・BEを問わず1つのコマンドで実装）

・FE/BE完全分業 → /fe-impl + /be-impl を使用
  （FE担当・BE担当が明確に分かれているチーム向け）
```

選択に応じて以下を生成する（既にファイルが存在する場合は上書き確認をする）。

| 選択 | 生成するファイル |
|------|----------------|
| フルスタック・分業なし | `.claude/skills/implement/conventions.md` |
| FE/BE完全分業 | `.claude/skills/fe-impl/conventions.md` + `.claude/skills/be-impl/conventions.md` |

生成する規約ファイルは雛形のため、プロジェクト固有のルールを後から追記することを案内する。

### 11. 完了報告

```
📊 コードベース分析完了
═══════════════════════════

📁 出力先: <ベースディレクトリ>/

📄 仕様書:
  - <ベースディレクトリ>/specs/<name>-spec.md
  - ...

📐 設計書:
  - <ベースディレクトリ>/designs/<name>-design.md
  - ...

📝 ナレッジ:
  - <ベースディレクトリ>/knowledge/<topic>.md
  - ...

これらのドキュメントにより、以下のスキルが利用可能になります:
  /implement（または /fe-impl, /be-impl） - 設計書に沿った実装
  /fix                                    - 仕様に基づくバグ修正
  /review                                 - 設計・仕様との整合性チェック

📋 CLAUDE.md:
  - [生成 or 更新] 済み

📝 規約ファイル:
  - [生成したファイルパス]（内容を確認・編集してください）

⚠️ 生成されたドキュメントはコードの現状を反映しています。
   内容を確認し、必要に応じて修正してください。

💡 その他のカスタマイズ
─────────────────────────
■ ドキュメントテンプレートがある場合:
  .claude/skills/analyze-codebase/templates/spec.md   - 仕様書テンプレート
  .claude/skills/analyze-codebase/templates/design.md  - 設計書テンプレート

■ プロジェクト共通ルール:
  .claude/rules/ 配下にMarkdownファイルを配置
  （全スキル・全操作で常に適用されます）
```

## 注意事項

- 生成するドキュメントはあくまで**コードの現状の記録**であり、理想像ではない
- 推測や補完は行わず、コードから読み取れる事実のみを記述する
- 大規模なコードベースの場合は、ユーザーと相談してスコープを絞る
- 既存のドキュメントがある場合は上書きせず、ユーザーに確認する
