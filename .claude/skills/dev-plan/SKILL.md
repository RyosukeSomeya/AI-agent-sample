# タスク分割・開発計画スキル

仕様書と詳細設計書を読み込み、PRのレビューしやすい粒度を考慮した最適なタスク分割と開発計画を生成する。

## 出力先

2種類のファイルを生成する。**いずれもgit管理対象外（ローカル運用）。** 実装者とClaudeの作業コンテキスト共有用であり、チームの進捗管理は外部チケットシステム（Backlog, Jira等）で行う。

### 開発計画ファイル

```
docs/plans/<plan-name>.md
```

### タスクファイル

```
docs/tasks/TASK-XXX.md
```

- `XXX`: 既存タスクの最大番号の続きから連番で振る（001, 002, ...）
- 既存タスクがある場合は既存の最大番号 + 1 から開始する
- 外部チケットIDがある場合はそれを使用してもよい

## 入力パターン

### パターン1: フルパス指定

```
/dev-plan docs/requirements/kaizen1-github-tree-requirements.md
```

### パターン2: 名前のみ指定

```
/dev-plan <name>
```

`docs/requirements/<name>-requirements.md` に解決する。

### パターン3: 引数なし

```
/dev-plan
```

`docs/requirements/` 配下の要件定義書を一覧し、対象を `AskUserQuestion` で確認する。

## 引数の解釈

`$ARGUMENTS` が `/` を含まない場合（名前のみ）、以下の順序でファイルを探す。

1. `docs/requirements/<name>-requirements.md` が存在すれば使用する
2. 存在しない場合は `docs/requirements/` 配下で名前を含むファイルを検索し、候補を提示する

## 前提確認

タスク分割開始前に以下を確認する。不足があれば該当スキルの実行を案内する。

1. `docs/specs/` に該当する仕様書が存在すること → なければ `/spec` を案内
2. `docs/designs/` に該当する設計書が存在すること → なければ `/design` を案内

## 入力ゲート

実行開始時に、入力となる仕様書と設計書のステータスが `✅ 承認済み` であることを確認する。
`📝 ドラフト` のドキュメントがある場合は、先にレビューを完了するよう案内して終了する。

> **例外:** ステータスヘッダーがない既存ドキュメントの場合は、ユーザーに確認の上、承認済みとして扱ってよい。

## 手順

1. 要件定義書を読み込む
   - 引数がファイルパスならそのファイルを `Read` で読み込む
   - 引数なしなら `docs/requirements/` 配下をスキャンし、対象をユーザーに選択してもらう
2. 対応する仕様書（`docs/specs/<feature-name>-spec.md`）を読み込む
3. 対応する設計書（`docs/designs/<feature-name>-design.md`）を読み込む
4. **入力ゲートを確認する**（仕様書・設計書が `✅ 承認済み` であること）
5. 既存のプラン・タスクファイルを確認し、新しいタスクの開始番号を決定する
6. **設計書の実装順序とコンポーネント構成を基に**、PR粒度を考慮してタスクを分割する
7. 開発計画ファイルを `docs/plans/<plan-name>.md` に保存する（ステータス: `📝 ドラフト`）
8. タスクファイルを `docs/tasks/TASK-XXX.md` として個別に保存する
9. **レビューサイクルを実行する**（後述の「レビューサイクル」セクション参照）

## タスク分割のルール

### 基本ルール

- **設計書の実装順序・コンポーネント構成を尊重する**
- 仕様書・設計書に書かれた機能のみをタスク化する。勝手に機能を追加しない
- タスク間の**依存関係を明確にする**（並行実行可能なタスクを識別する）
- DB変更 → ユーティリティ → コンポーネント → ページ統合 の順で依存を組む
- 各タスクの完了条件は設計書の確認方法を参考にする

### PR粒度の考慮（1タスク = 1PR を前提）

タスクの粒度は**人間がレビューしやすいPRサイズ**を基準に決定する。

- **差分の目安:** 1タスクあたり変更行数 **200行以下** を推奨する（論理的な区切りの都合上300行程度になるのは許容）
- **レビュー観点の統一:** 1タスクは1つの論理的な変更単位とする（例: 「型定義の追加」「APIエンドポイントの実装」「UIコンポーネントの作成」）
- **動作確認の容易さ:** 各タスク単体で動作確認可能な状態にする
- **大きなタスクの分割:** 設計書上の1ステップが大きすぎる場合はさらに分割する
  - 例: 「CRUD APIの実装」→「Create/Read APIの実装」+「Update/Delete APIの実装」
- **小さすぎるタスクの統合:** 単独ではレビューの意味が薄い変更は統合する
  - 例: 型定義のみ、定数定義のみ → 最初に使用するタスクに統合

### 分割の判断基準

| 分割する | 統合する |
|---------|---------|
| 変更ファイルが5つ以上になる | 変更ファイルが1つで数行のみ |
| 異なるレイヤー（DB/API/UI）にまたがる | 同一ファイル内の関連する変更 |
| 別の人が並行で作業できる | 前のタスクの直後でしか意味がない微小変更 |

## 開発計画ファイルのテンプレート

```markdown
# <計画名>

**作成日:** <生成日>
**ステータス:** 📝 ドラフト
**要件定義書:** [<ファイル名>](../requirements/<ファイル名>)
**仕様書:** [<ファイル名>](../specs/<ファイル名>)
**設計書:** [<ファイル名>](../designs/<ファイル名>)

---

## タスク一覧

| ID       | タスク名                    | 依存タスク | 想定PR規模 | ステータス  |
| -------- | --------------------------- | ---------- | ---------- | ----------- |
| TASK-XXX | <タスク名>                  | -          | S/M/L      | ⬜ 未着手   |

> PR規模目安: S(〜100行) / M(100〜200行) / L(200行〜、要注意)

---

## タスク概要

### TASK-XXX: <タスク名>

<概要>
- 実装内容

---

## 依存関係図

（Mermaid図）

---

## 推奨実装順序

依存関係に基づく実装順序。
```

## タスクファイルのテンプレート

```markdown
# TASK-XXX: <タスク名>

**ステータス:** ⬜ 未着手
**依存タスク:** <依存タスクID or なし>
**設計書:** [<ファイル名>](../../docs/designs/<ファイル名>)
**想定PR規模:** S/M/L

## 概要

<このタスクで何を実現するか>

## スコープ

- 実装項目（設計書の該当セクション参照）

## 完了条件

- <動作確認可能な具体的な条件>
- `yarn build` と `yarn lint` がエラーなく通ること
```

## レビューサイクル

### 1. レビューポイントの提示

生成した開発計画の主要な内容を箇条書きで提示し、`AskUserQuestion` で承認を求める。

```
📋 開発計画のレビュー
─────────────────────
プラン: docs/plans/<plan-name>.md
タスク: docs/tasks/TASK-XXX.md 〜 TASK-YYY.md

【タスク一覧】
- TASK-XXX: <タスク名> (規模: M)
- TASK-YYY: <タスク名> (規模: S)
- ...

【依存関係】
- <依存関係の概要>

【推奨実装順序】
1. TASK-XXX → 2. TASK-YYY → ...

この内容で承認しますか？
```

選択肢: 「承認する」「修正を依頼する」「再生成する」

### 2. 修正ループ（修正依頼の場合）

1. ユーザーのフィードバック内容を確認する
2. プラン・タスクファイルを修正する
3. 修正箇所の差分を提示する
4. 再度 `AskUserQuestion` で承認を求める

### 3. 承認後の処理

1. プランのステータスを `✅ 承認済み` に更新する
2. 次のステップを案内する

## 完了後の案内

レビュー承認後、以下のように案内する:

```
✅ 開発計画を承認しました: docs/plans/<plan-name>.md
タスクファイル: docs/tasks/TASK-XXX.md 〜 TASK-YYY.md

最初のタスクの実装を開始できます:

  /implement TASK-XXX
```

## $ARGUMENTS

- `/dev-plan` - 要件定義書を選択してタスク分割
- `/dev-plan docs/requirements/<name>-requirements.md` - 指定した要件定義書からタスク分割
