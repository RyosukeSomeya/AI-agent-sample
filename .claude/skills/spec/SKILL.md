# 仕様書生成スキル

要件定義書を読み込み、機能全体の仕様書を1ファイルとして生成する。

## 出力先

`docs/specs/` ディレクトリに、要件定義書単位で1ファイルとして出力する。

### ファイル命名規則

```
docs/specs/<feature-name>-spec.md
```

- `<feature-name>`: 要件定義書のプロジェクト名に対応するkebab-case名
- 例: `docs/requirements/kaizen1-github-tree-requirements.md` → `docs/specs/kaizen1-github-tree-spec.md`

## 入力パターン

- `/spec docs/requirements/<name>-requirements.md` - フルパス指定
- `/spec <name>` - 名前のみ指定（`docs/requirements/<name>-requirements.md` に解決）
- `/spec` - `docs/requirements/` 配下の要件定義書を一覧し、対象を選択して生成

## 引数の解釈

`$ARGUMENTS` が `/` を含まない場合（名前のみ）、以下の順序でファイルを探す。

1. `docs/requirements/<name>-requirements.md` が存在すれば使用する
2. 存在しない場合は `docs/requirements/` 配下で名前を含むファイルを検索し、候補を提示する

## 入力ゲート

実行開始時に、入力となる要件定義書のステータスが `✅ 承認済み` であることを確認する。
`📝 ドラフト` の場合は、先にレビューを完了するよう案内して終了する。

> **例外:** ステータスヘッダーがない既存ドキュメントの場合は、ユーザーに確認の上、承認済みとして扱ってよい。

## 手順

1. 要件定義書を `Read` で読み込む（引数なしの場合は `docs/requirements/` をスキャンして選択）
2. **入力ゲートを確認する**（要件定義書が `✅ 承認済み` であること）
3. 既存コードベースを確認し、現状の実装を把握する
4. 要件定義書に含まれる**すべての機能要件**について画面・API・データモデルの仕様を記述する
5. `docs/specs/<feature-name>-spec.md` に保存する（ステータス: `📝 ドラフト`）
6. **レビューサイクルを実行する**（後述の「レビューサイクル」セクション参照）

## 生成ルール

- 要件定義に書かれていない機能を追加しない
- 曖昧な要件は「要確認」として明示し、仮の仕様を併記する
- 各画面・各機能の入力・出力・振る舞いを具体的に記述する
- エッジケース（空データ、エラー時、境界値）を記述する
- 既存コードがある場合は現行の実装との整合性を確認する
- **図解にはMermaid記法を使用する**（画面遷移図、シーケンス図、ER図、状態遷移図等）

## 仕様書の構成

```markdown
# <機能名> 仕様書

**要件定義書:** [<ファイル名>](../requirements/<ファイル名>)
**作成日:** <生成日>
**ステータス:** 📝 ドラフト

---

## 1. 画面仕様

### 画面遷移図
（Mermaid記法で記述）

### <画面名>

#### 表示要素
- 各UI要素の一覧

#### 初期状態
- 画面を開いたときのデフォルト値、データ取得処理

#### ユーザー操作と振る舞い
- 操作ごとの挙動
- バリデーションルール
- 成功時・失敗時の挙動
（必要に応じてMermaid シーケンス図で操作フローを図示）

#### エラーハンドリング
- エラーメッセージとその条件

## 2. API仕様

### <エンドポイント名>

| 項目 | 内容 |
|------|------|
| メソッド・パス | GET /api/xxx |
| リクエスト | パラメータ、ボディの型 |
| レスポンス | 成功時・エラー時の型 |
| 処理概要 | 何をするか |

## 3. データモデル

### ER図
（Mermaid erDiagram で記述）

各テーブルのフィールド、型、制約。
要件定義にDB設計がある場合はそれを踏襲する。

## 4. 要確認事項

要件定義から読み取れなかった点をリストアップする。
```

## レビューサイクル

`development-cycle.md` の「レビューサイクル（全フェーズ共通）」に従い、以下を実行する。

### 1. レビューポイントの提示

生成した仕様書の主要な内容を箇条書きで提示し、`AskUserQuestion` で承認を求める。

```
📋 仕様書のレビュー
─────────────────────
ファイル: docs/specs/<feature-name>-spec.md

【主要な内容】
- 画面仕様: <画面数と概要>
- API仕様: <エンドポイント数と概要>
- データモデル: <テーブル/変更の概要>
- 要確認事項: <件数と内容>

この内容で承認しますか？
```

選択肢: 「承認する」「修正を依頼する」「再生成する」

### 2. 修正ループ（修正依頼の場合）

1. ユーザーのフィードバック内容を確認する
2. ドキュメントを修正する
3. 修正箇所の差分を提示する
4. 再度 `AskUserQuestion` で承認を求める

### 3. 承認後の処理

1. ドキュメントのステータスを `✅ 承認済み` に更新する
2. 次のステップを案内する

## 完了後の案内

レビュー承認後、以下のように案内する:

```
✅ 仕様書を承認しました: docs/specs/<feature-name>-spec.md

次のコマンドで詳細設計書を作成できます:

  /design docs/specs/<feature-name>-spec.md
```
