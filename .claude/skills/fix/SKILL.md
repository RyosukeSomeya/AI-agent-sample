# 修正スキル

不具合を修正し、動作確認で問題解消を確認した後、必要に応じて仕様書・設計書を更新する。

## 入力

- `$ARGUMENTS`: 修正内容の説明、またはエラー分析ファイルのパス（例: `docs/analysis/ERR-001.md`）

## 引数の解釈

`$ARGUMENTS` がエラーID・ファイル名のみ（`/` を含まない）の場合、以下の順序でファイルを探す。

1. `docs/analysis/<name>.md` が存在すれば使用する
2. 存在しない場合は `docs/analysis/` 配下で名前を含むファイルを検索し、候補を提示する
3. 該当なしの場合は問題の説明として扱う

## ファイアウォール制約

`.claude/rules/firewall.md` のルールに従うこと。

## 修正フロー

```
入力（問題の説明 or 分析ファイル）
    │
    ▼
1. 調査・原因特定
    │
    ▼
2. 修正方針をユーザーに提示 → 承認
    │
    ▼
3. 修正を実装
    │
    ▼
4. 動作確認（ビルド・リント + 問題の再現確認）
    │
    ├── NG → 1に戻る（再調査。何がNGかを明示する）
    │
    ▼  OK
5. ドキュメント更新（仕様/設計に影響がある場合）
    │
    ▼
6. 完了報告 → /commit へ
```

## 手順

### 1. 調査・原因特定

- $ARGUMENTS がファイルパス（docs/analysis/ERR-XXX.md）の場合、分析レポートを読み込む
- $ARGUMENTS が説明文の場合、問題の内容を把握しコードを調査する
- `docs/specs/` の該当仕様書と `docs/designs/` の該当設計書を参照し、正しい挙動を確認する
- 原因を特定する

### 2. 修正方針の提示

```
🔍 問題分析
─────────────────────
原因: [原因の説明]
影響範囲: [影響を受けるファイル・機能]
関連タスク: TASK-XXX（該当する場合）

📋 修正方針
─────────────────────
1. [修正内容1]
2. [修正内容2]

⚠️ ドキュメント更新
─────────────────────
- 仕様書(docs/specs/): [更新の要否と内容]
- 設計書(docs/designs/): [更新の要否と内容]

この方針で修正してよいですか？
```

### 3. 修正の実装

修正前に規約ファイルを読み込む。

- `.claude/skills/implement/conventions.md` が存在すれば読み込む
- 存在しない場合は `.claude/skills/fe-impl/conventions.md` と `.claude/skills/be-impl/conventions.md` を確認し、存在するものを読み込む
- FEファイル（コンポーネント、ページ、hooks等）はFE規約を、BEファイル（API、DB、サービス層等）はBE規約を適用する

規約を読み込んだ上で、承認された方針に従って修正する。修正範囲は最小限に保つ。

### 4. 動作確認

```bash
yarn build
yarn lint
```

加えて、修正対象の問題が解消されたことの確認方法をユーザーに提示する。

**NGの場合**: 何がNGだったかを明示し、ステップ1に戻る。再調査の際は前回の修正で何が起きたかを踏まえて分析する。

### 5. ドキュメント整合性チェックと更新

動作確認OKの場合、**必ず**仕様書・設計書との整合性を確認し、必要に応じて更新する。**このステップは省略不可。**

#### 確認手順（doc-checker エージェントで並列実行）

カスタムサブエージェント `doc-checker`（`.claude/agents/doc-checker.md` に定義済み）を `Agent` ツールで呼び出す。
設計書チェックと仕様書チェックを**2つ並列に**実行する。

```
エージェント①: 「以下の修正ファイルと設計書を比較してください。
 修正ファイル: [修正したファイルパスの一覧]
 設計書: docs/designs/<feature-name>-design.md」

エージェント②: 「以下の修正ファイルと仕様書を比較してください。
 修正ファイル: [修正したファイルパスの一覧]
 仕様書: docs/specs/<feature-name>-spec.md」
```

両方の結果を受け取り、差異がある場合は以下の判断基準で更新する。

#### 更新の判断基準

| 条件 | 更新対象 |
|------|---------|
| 仕様の誤りが原因 | `docs/specs/` の該当仕様書を修正 |
| 設計の誤りが原因 | `docs/designs/` の該当設計書を修正 |
| 仕様・設計は正しく実装バグ | 更新不要（ただし整合性は確認すること） |
| 新しいエッジケースが判明 | `docs/specs/` の該当仕様書に追記 |
| 修正により関数名・型・APIの振る舞いが変わった | 設計書・仕様書の両方を更新 |

更新箇所には以下のコメントを残す。

```markdown
<!-- fix: YYYY-MM-DD [修正内容の要約] -->
```

#### 完了報告にドキュメント更新の有無を含める

更新した場合はファイル名と変更内容を、更新不要だった場合は「整合性確認済み・更新不要」と報告する。

### 6. ナレッジの記録

修正を通じて得られた知見（環境固有の制約、パッケージの注意点、ハマりどころ等）があれば `docs/knowledge/` にMarkdownファイルとして記録する。

### 7. 完了報告

```
✅ 修正完了
─────────────────────
修正内容: [何を修正したか]
原因: [何が原因だったか]
修正ファイル:
  - [ファイルパス] ([変更概要])
ドキュメント更新:
  - [更新内容、または「なし」]
─────────────────────

コミットするには:
  /commit
```

## $ARGUMENTS

- `/fix ボタンをクリックしても画面遷移しない` - 問題の説明から修正
- `/fix docs/analysis/ERR-001.md` - エラー分析レポートを入力として修正
