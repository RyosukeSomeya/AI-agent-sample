# バックエンド実装スキル

詳細設計書とBEコーディング規約に従ってバックエンド実装を行う。

> **このスキルの用途:** FE担当・BE担当で完全に分業するチームのBE担当者向け。フルスタックや分業なしのチームは `/implement` を使用する。

## 入力パターン

- `/be-impl TASK-XXX` - 指定タスクのスコープに沿って実装（推奨）
- `/be-impl` - 未実装の次のタスクを自動検出して実装

## 前提確認

実装開始前に以下を確認する。不足があれば該当スキルの実行を案内する。

1. `docs/requirements/` 配下に要件定義書が存在すること
2. `docs/specs/` 配下に該当機能の仕様書が存在すること
3. `docs/designs/` 配下に該当機能の設計書が存在すること
4. `docs/tasks/TASK-XXX.md` が存在し、スコープと完了条件が定義されていること

## 入力ゲート

実行開始時に、該当する開発計画（`docs/plans/`）のステータスが `✅ 承認済み` であることを確認する。
`📝 ドラフト` の場合は、先にレビューを完了するよう案内して終了する。

> **例外:** ステータスヘッダーがない既存ドキュメントの場合は、ユーザーに確認の上、承認済みとして扱ってよい。

## ファイアウォール制約

`.claude/rules/firewall.md` のルールに従うこと。

## 手順

### 1. BE規約の読み込み

スキルディレクトリ内の `conventions.md` を `Read` で読み込み、実装中はこれに従う。

### 2. タスクと設計の確認

1. `docs/tasks/TASK-XXX.md` を読み、スコープと完了条件を確認する
2. タスクファイルに記載された設計書のリンクから設計書を読む
3. 設計書の該当セクション（API設計、DB設計、サービス層設計）を特定する
4. `docs/knowledge/` を確認し、実装に関連するナレッジがあれば参照する
5. `docs/plans/` 内の該当プランのステータスを `🔨 実装中` に更新する

### 3. 実装

- **設計書に記載されたディレクトリ構成、ファイルパス、関数名をそのまま使用する**
- 設計書にないファイルを勝手に作成しない
- 設計書にない機能を勝手に追加しない
- 設計と異なる判断が必要な場合は、実装を止めてユーザーに確認する
- TypeScriptの型を厳密に定義する（anyを使わない）
- 仕様書に記載されたバリデーションルール・エラーハンドリングを漏れなく実装する

### 4. 各ステップの確認

各実装ステップ完了時に以下を実行する。

```bash
yarn build
yarn lint
```

エラーがあれば修正してから次のステップに進む。

### 5. 設計書との差異が発生した場合

1. **実装を一旦止める**
2. 何が問題で、どう変更すべきかをユーザーに報告する
3. ユーザーの承認を得てから変更する
4. `docs/designs/` の該当設計書にも変更を反映する
5. 必要であれば `docs/specs/` の該当仕様書も更新する

### 6. ドキュメント整合性チェック

実装完了後、コードと仕様書・設計書が一致しているかを確認する。**このステップは必須。**

カスタムサブエージェント `doc-checker`（`.claude/agents/doc-checker.md` に定義済み）を `Agent` ツールで呼び出す。
設計書チェックと仕様書チェックを**2つ並列に**実行する。

差異が見つかった場合は更新箇所に以下のコメントを残す:

```markdown
<!-- implement: YYYY-MM-DD TASK-XXX [変更内容の要約] -->
```

### 7. ナレッジの蓄積

実装中に得た知見（環境固有の制約、パッケージの注意点、ハマりどころ等）があれば `docs/knowledge/` に記録する。

## 完了後の状態

- タスクのスコープが実装されている
- ビルドエラー・リントエラーがない
- BE規約に準拠している
- 設計書にない余計なファイル・機能が追加されていない
- 仕様書・設計書が実装後のコードと一致している
- 実装中に得た知見があれば `docs/knowledge/` に記録されている

## $ARGUMENTS

- `/be-impl TASK-XXX` - 指定タスクのスコープに沿ってBE実装
- `/be-impl` - 次の未実装タスクを自動検出してBE実装
