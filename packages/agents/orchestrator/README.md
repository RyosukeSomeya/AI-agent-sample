# オーケストレータ（orchestrator）

## 学習テーマ

A2A（Agent-to-Agent）連携の核心パターンを学ぶ。Agent インスタンスをツールとして別の Agent に渡すだけで、エージェント間の協調が実現する。

- **対応ワークショップ:** ワークショップ外 / 発展（マルチエージェント設計）
- **対応Step:** Step 3
- **対応タスク:** TASK-006

## このエージェントの役割

4つの専門エージェント（収集・分析・横断分析・異常検知）を統合し、ユーザーの指示に応じて適切なエージェントに仕事を振り分ける「チームリーダー」。

### 学習ポイント

1. **A2A連携の実装**: `Agent` インスタンスを `tools` リストに渡すだけで、エージェント間通信が実現する
2. **LLM動的判断**: オーケストレータのLLMが「どのエージェントに何を頼むか」を自然言語で判断する
3. **ハイブリッド方式の内側**: 本番では Step Functions（外側）+ A2A（内側）の二層構造。このオーケストレータは内側に相当

### エージェント連携フロー

```
ユーザー: "東京・大阪・福岡の天気を比較して"
  │
  ▼
オーケストレータ（LLMが判断）
  ├─→ collector: 東京のデータ取得
  ├─→ collector: 大阪のデータ取得
  ├─→ collector: 福岡のデータ取得
  ├─→ analyst: 各都市の個別分析
  ├─→ crosscut: 3都市の横断比較
  └─→ alert: 異常があればアラート生成
  │
  ▼
統合レポートをユーザーに返す
```

## 動作確認手順

```bash
cd packages/agents
uv run python -m orchestrator
```

起動後、以下のようなプロンプトを入力する。

例:
```
東京・大阪・福岡の天気を比較分析してください
```

## 本番構成との差分

| 項目 | サンプル | 本番 |
|------|---------|------|
| 外側の制御 | なし（オーケストレータが全て担当） | Step Functions（決定的制御） |
| 内側の判断 | オーケストレータ Agent（LLM動的判断） | AgentCore A2A（LLM動的判断） |
| エージェント呼び出し | 同一プロセス内で直接呼び出し | AgentCore Runtime（HTTPS / microVM） |
| 並列実行 | LLMが逐次判断 | Step Functions Map State で並列実行 |
